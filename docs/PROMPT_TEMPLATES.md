# Prompt Templates

## BACKLOG文言調整プロンプト（完全版 / 保存用）

```text
あなたはシニアエンジニア兼監査ドキュメント編集責任者です。以下の要件と制約に従って、docs/BACKLOG.md を編集してください。

**絶対に推測で仕様を決定しないでください。要件に不足がある場合は、コードを書く前に必ず質問してください。**

<objective>

docs/BACKLOG.md を、VPS再監査結果を反映済みの **SSOT（Single Source of Truth）文書**として、**意味・判定を一切変えずに** 表現だけ最終調整する。

今回の目的は **追加監査ではなく文書精度の最終仕上げ** である。

</objective>

<context>

- BACKLOG.md の再監査自体は完了済み。
- VPS実態に基づく [x] / [ ] 判定はすでに確定している。
- 今回はその監査結果の意味を変えず、文面だけを「実装準拠で誤解のない表現」に整える。
- 特に H3 と K3 は、読んだ人が誤解しやすい表現になりやすいため重点確認対象。
- 監査済み事実は維持し、文書上の主語・条件・時点を明確化することが主眼。

</context>

<tech_stack>

- 対象ファイル: `docs/BACKLOG.md`
- 作業内容: Markdownドキュメント編集
- Git運用: 変更があれば commit / push する
- 追加のVPS監査・SSH確認・新規実装は不要

</tech_stack>

<scope>

- 対象: `docs/BACKLOG.md`
- 対象外: VPS/SSH/cron/systemd/DB/新規実装/新規監査

</scope>

<idempotency_rule>

この作業は冪等に扱うこと。
まず `git log --oneline -3` と `git diff -- docs/BACKLOG.md` を確認し、直近の同趣旨 commit または未 commit の同趣旨変更があれば再編集せず **「追加修正なし」** で終了する。

</idempotency_rule>

<workflow>

1. `git log --oneline -3` と `git diff -- docs/BACKLOG.md` を実行し、直近の commit 目的と未 commit 変更の両方を確認してから作業要否を判断する。
2. `docs/BACKLOG.md` を読む。
3. H3 を確認する。
4. K3 を確認する。
5. 文書全体の「実態より強い表現」を確認する。
6. 必要時のみ最小差分で編集する。
7. 変更があれば diff 確認 → commit → push。
8. 変更不要なら無変更で終了する。

</workflow>

<requirements>

- `docs/BACKLOG.md` を読み、表現精度の観点で最終調整する。
- **H3** について、本文主語が `handoff.json` の常時存在を前提に読めるなら、以下の方向に修正すること:
  - `session-end.py` に **handoff保存ロジックが実装済み**
  - `session-start.sh` に **handoff読み込みロジックが実装済み**
  - `handoff.json` は **runtime artifact** であり、**条件付き生成** である
- **K3** について、以下を混同しない文面にすること:
  - **K3完了時点で何を達成したか**
  - **その後の運用上のBrier Score変動**
- 文書全体を見て、以下のような実態より強い表現が残っていれば弱めること:
  - 「常時存在」
  - 「常時稼働」
  - 「常に生成される」
  - 「必ず存在」
  - 条件付きなのに無条件に見える表現
- ただし、意味を変えず、監査済み事実の範囲でのみ調整すること。
- 修正後、変更差分を確認し、必要なら git commit / push すること。
- 変更不要ならファイルは触らず、「追加修正なし」で終了すること。

</requirements>

<constraints>

- **[x] / [ ] 判定は変更しないこと。**
- 判定変更は、明白な誤記・矛盾が本文中にあり、修正しないと文書自体が破綻する場合のみ検討可。ただし今回は原則変更禁止。
- **追加のVPS監査は禁止。**
- **新しい事実の追加は禁止。**
- 既存の監査結果・注記の意味を変える言い換えは禁止。
- 文書の目的は「読み手が誤解しないこと」であり、「強く見せること」ではない。
- runtime artifact は、**ファイルの恒常的存在**ではなく、**ロジック実装 + 条件付き生成** として表現すること。
- K3 の Brier に関しては、**K3実装そのものの成否** と **運用後の成績変動** を分離して書くこと。
- 不要な大改稿は禁止。必要最小限の差分で仕上げること。
- 推測による補足説明や拡大解釈は禁止。

</constraints>

<edge_cases_and_error_handling>

- H3 において `handoff.json` が存在しないタイミングがあっても、それは未実装を意味しない。条件付き生成の実装であることを正確に表現する。
- K3 において Brier Score が後日悪化していても、それを K3 実装失敗と誤読される表現は避ける。
- 「保存される」「読み込まれる」などの受動表現が無条件に見える場合は、条件を明示する。
- 修正候補があるが意味変更リスクが高い場合は、無理に書き換えず最小修正に留める。
- 変更が本当に不要なら、無変更で終了する。

</edge_cases_and_error_handling>

<output_format>

- まず `<thinking>` タグ内で、どの表現が誤解を生みうるか、どこをどう直すべきかを簡潔に整理すること。
- その後、以下の順で出力すること。

1. 修正対象
2. 修正前→修正後の要点
3. 変更理由
4. git commit hash（変更時のみ）
5. 最終結論

- 実際にファイルを修正した場合は、差分が分かるように説明すること。
- コードや文章は省略せず、完全に動作する状態・実行済み状態で報告すること。
- 「追加修正なし」の場合も、その根拠を明記すること。
- 同じ結論を重複して書かないこと。

</output_format>
```

あなたはシニアエンジニア兼監査ドキュメント編集責任者です。`docs/BACKLOG.md` の**文言だけ**を最終調整してください。

<critical_rule>
今回の作業は追加監査ではない。VPS/SSH/cron/systemd/DB確認、新規実装、新規監査は禁止。

<scope>

</scope>

<idempotency_rule>

2. `docs/BACKLOG.md` を読む

- `handoff.json` は runtime artifact であり条件付き生成である

- 常時稼働

大改稿禁止。必要最小限の差分のみ。
受動表現で無条件に見える場合は条件を明示する。
意味変更リスクが高い箇所は無理に広く直さない。
</constraints>

<output_format>
最初に `<thinking>` で以下を簡潔に書くこと:
- 誤読リスクのある箇所
- 修正要否
- 既修正なら無変更でよい理由

その後、必ずこの順で出力すること:
1. 修正対象
2. 修正前→修正後の要点
3. 変更理由
4. git commit hash（変更時のみ）
5. 最終結論
