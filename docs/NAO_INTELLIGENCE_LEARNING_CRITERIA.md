# Nao Intelligence — 学習基準（Track 5）
> 作成: 2026-03-26 | セッション継続（完全書き直し）
> 目的: 「学習が起きた」を9つのタイプ別に、測定可能な形で定義する。
> 原則: 事実・仮定・提案を分離する。「気がする」は学習ではない。

---

## 定義の構造

各学習タイプについて、以下の3つを明確にする:

| 項目 | 意味 |
|------|------|
| **何が記録されるか** | 学習が起きた際に永続ストレージに書かれるもの |
| **何をもって更新とするか** | 「更新した」と判断できる客観的条件 |
| **どう検証するか** | 後から「本当に学習したか」を確認する方法 |

---

## タイプ1: 新しい事実の取得（New Fact Acquisition）

**定義**: 今まで知らなかった外部事実がシステムに取り込まれ、次回以降に参照可能になった状態。

| 項目 | 内容 |
|------|------|
| **何が記録されるか** | `AGENT_WISDOM.md` の学習ログセクションに「事実: [内容] / 取得日: / 情報源:」の形式で追記。Polymarketデータならcategory_brier.json経由で自動更新。 |
| **何をもって更新とするか** | ファイルの更新タイムスタンプが変わり、新規エントリが1件以上追加されている。単なる「処理した」ではなく、具体的な事実文が書かれていること。 |
| **どう検証するか** | `grep -c "事実:" /opt/shared/AGENT_WISDOM.md` の件数が前回比で増加。または `cat /opt/shared/observer_log/YYYY-MM-DD.json` に当日の新規事実エントリが存在する。 |

---

## タイプ2: 誤りの訂正（Error Correction）

**定義**: 過去の誤った情報・判断・コードが特定され、正しいものに置き換わり、再発防止策が記録された状態。

| 項目 | 内容 |
|------|------|
| **何が記録されるか** | `docs/KNOWN_MISTAKES.md` に「症状 / 根本原因 / 正しい解決策 / 教訓 / 再発防止コード」の5フィールドで記録。`mistake_patterns.json` にパターンが登録される。 |
| **何をもって更新とするか** | KNOWN_MISTAKES.mdに新規エントリが追加 **かつ** auto-codifier.pyが mistake_patterns.json を更新した。ドキュメントだけでパターン登録がないものは「未完了」とみなす。 |
| **どう検証するか** | `python .claude/hooks/regression-runner.py` で全25テストがPASS。同カテゴリのミスが30日間再発していないことを `grep` で確認。 |

---

## タイプ3: 判断の改善（Judgment Improvement）

**定義**: 同じ種類の意思決定において、より良い選択を行えるようになった状態。確率推定・リスク評価・優先順位付けの精度向上を含む。

| 項目 | 内容 |
|------|------|
| **何が記録されるか** | `AGENT_WISDOM.md` の「判断ルール」セクションに「[ドメイン]において[条件]なら[判断]が優れている — 根拠: [データ]」の形式で記録。Reflexion出力がある場合は `/opt/shared/observer_log/` に構造化保存。 |
| **何をもって更新とするか** | 新しい判断ルールが1件以上追記 **かつ** その根拠として具体的な数値（Brier Score、エラー率、成功率）が示されている。「〜と感じた」だけのエントリは更新とみなさない。 |
| **どう検証するか** | 同種の判断タスクをBefore/Afterで比較。予測タスクならBrier Score週次比較（`evolution_loop.py` が自動生成）。判断エラーなら KNOWN_MISTAKES の同カテゴリ再発件数。 |

---

## タイプ4: 実行の改善（Execution Improvement）

**定義**: 同じタスクをより少ないステップ・より少ないエラー・より高い完成度で実行できるようになった状態。

| 項目 | 内容 |
|------|------|
| **何が記録されるか** | タスク完了後に `/opt/shared/task-log/YYYY-MM-DD_agent_タスク名.md` に「実行ステップ数 / エラー発生件数 / 修正回数 / 完了品質スコア」を記録。繰り返しタスクは前回比で比較可能。 |
| **何をもって更新とするか** | 同一タスクカテゴリで2回以上の実行履歴があり、最新実行の「エラー発生件数」または「修正回数」が前回より減少している。または新しい効率化手順が AGENT_WISDOM に追記されている。 |
| **どう検証するか** | `ls /opt/shared/task-log/ | grep [タスクカテゴリ]` で過去ログを取得し、エラー件数推移を比較。記事生成なら QA Sentinel の品質スコア推移（`article_health_db.sqlite`）。 |

---

## タイプ5: 反省の保存（Reflection Saving）

**定義**: セッション内で発生した「気づき・反省・学び」が、セッション終了後も参照可能なストレージに書き込まれた状態。セッション記憶の持続化。

| 項目 | 内容 |
|------|------|
| **何が記録されるか** | `/opt/shared/observer_log/YYYY-MM-DD.json` に `{"type": "reflection", "content": "...", "agent": "...", "ts": "..."}` 形式で追記。MEMORY.md にはセッションサマリーとして要約版が書かれる。 |
| **何をもって更新とするか** | セッション終了時に `observer_log/` の当日ファイルにリフレクションエントリが1件以上存在する。単なるタスクログではなく「なぜそうしたか / 何がうまくいかなかったか」の説明文が含まれること。 |
| **どう検証するか** | `cat /opt/shared/observer_log/YYYY-MM-DD.json | python3 -c "import json,sys; data=json.load(sys.stdin); print(len([e for e in data if e.get('type')=='reflection']))"` で件数確認。0件なら保存失敗。 |

---

## タイプ6: 次回セッションへの反省の引き継ぎ（Cross-Session Reflection Transfer）

**定義**: 前回セッションの反省が、次回セッション開始時のコンテキストに実際に注入され、行動に影響を与えた状態。「保存」（タイプ5）と「活用」は別物。

| 項目 | 内容 |
|------|------|
| **何が記録されるか** | `session-start.sh` が `observer_log/` の直近エントリを読み込み、MEMORY.md の「前回からの引き継ぎ」セクションに注入した内容。引き継ぎに成功した場合は `SHARED_STATE.md` の「セッション引き継ぎ精度」フィールドを更新。 |
| **何をもって更新とするか** | 次回セッションの最初の行動が、前回の反省内容と一致する判断をしている。例: 前回「ENタグ確認を忘れた」→ 今回セッション冒頭で ENタグ確認を自発的に行った。 |
| **どう検証するか** | （現状）人間が判断（Naotoが「前回の反省が活きているか」を確認）。（Phase 2以降）Mem0経由でセッション引き継ぎ精度を定量化。目標: 54% → 80%。 |

---

## タイプ7: 予測精度の改善（Prediction Accuracy Improvement）

**定義**: 特定カテゴリの予測において、Brier Scoreが統計的に有意に改善した状態。「1件当たった」は学習の証拠ではない。

| 項目 | 内容 |
|------|------|
| **何が記録されるか** | `evolution_loop.py` が毎週日曜に `category_brier.json` を更新し、`evolution_log.json` に週次比較エントリを追記。外れた予測には `root_cause` フィールドが記録される。 |
| **何をもって更新とするか** | 同カテゴリで5件以上の解決済み予測があり、直近4週のBrier Score平均が前4週比で0.01以上改善。または特定の弱点カテゴリ（経済・貿易: 0.4868, 暗号資産: 0.3334）で明確な改善傾向。 |
| **どう検証するか** | `cat /opt/shared/logs/category_brier.json` でカテゴリ別Brier Score推移を確認。`evolution_log.json` で「Brier Score改善」エントリが存在するか確認。目標: 総合0.1776 → 0.15。 |

---

## タイプ8: 長期的な価値判断の更新（Long-term Value Judgment Update）

**定義**: 何が「良い判断か」「重要なトピックか」「信頼できる情報源か」という基準そのものが更新された状態。戦術ではなく価値基準の変化。

| 項目 | 内容 |
|------|------|
| **何が記録されるか** | `AGENT_WISDOM.md` の「価値判断ルール」セクションに「[ドメイン]において[古い基準]は[新しい基準]に更新された — 証拠: [具体的データ] — 更新日:」の形式で記録。Reflectorエージェントが月次で統合。 |
| **何をもって更新とするか** | 以前の価値判断ルールが明示的に「廃止」または「更新」と記録され、新しいルールが具体的な証拠（データ・事例）と共に追記されている。証拠なしの更新は不可。 |
| **どう検証するか** | AGENT_WISDOM.md の「価値判断ルール」セクションに更新エントリが存在し、古いルールとの差分が明確。月次レビューでNaotoが「この判断変化は妥当か」を確認（C型: 人間判断）。 |

---

## タイプ9: オペレーターの意図との整合性向上（Operator Intent Alignment Improvement）

**定義**: Naotoが意図する方向（NORTH_STAR、Nowpatternのミッション）と、AIの実際の行動・提案の一致度が高まった状態。

| 項目 | 内容 |
|------|------|
| **何が記録されるか** | `MEMORY.md` の「オーナー優先事項」セクションに、Naotoが明示的に述べた意図・優先順位・制約を記録。毎セッションで「今回Naotoが強調したこと」を1〜3行で更新。 |
| **何をもって更新とするか** | AIが提案した内容のうち、Naotoに却下された提案が前月比で減少している。または「こういう理解でいいですか？」確認なしに実装を始めてNaotoに訂正された件数がゼロ。 |
| **どう検証するか** | （現状）Naotoが主観評価（「最近のAIは意図を汲んでいる」）。（計測可能版）提案却下率 = rejected / (approved + rejected) を月次で追跡。現在値: 未計測 → 目標: < 10%。 |

## 学習証拠の強度マトリックス

| 証拠の種類 | 強度 | タイプ1-9 での該当 |
|-----------|------|------------------|
| Brier Score数値改善（週次比較） | ★★★★★ | タイプ7 |
| 同カテゴリエラー30日再発ゼロ | ★★★★☆ | タイプ2 |
| task-log でのエラー件数減少傾向 | ★★★★☆ | タイプ4 |
| observer_log にリフレクションエントリ存在 | ★★★☆☆ | タイプ5 |
| AGENT_WISDOM に根拠付き新ルール追記 | ★★★☆☆ | タイプ3, 8 |
| KNOWN_MISTAKES + パターン登録の両方完了 | ★★★☆☆ | タイプ2 |
| セッション引き継ぎで行動変化あり | ★★☆☆☆ | タイプ6 |
| Polymarket/外部データとの照合 | ★★☆☆☆ | タイプ1 |
| 提案却下率の低下 | ★★☆☆☆ | タイプ9 |
| 自己報告（「わかりました」「改善しました」） | ★☆☆☆☆ | 証拠にならない |

**原則: ★3以上の証拠がない「学習した」報告は受け付けない。**

## アンチパターン（学習に見えて学習でないもの）

| パターン | なぜダメか | 本来すべきこと |
|---------|-----------|--------------|
| タスク完了 = 学習と報告 | 完了しても次回に引き継がれなければゼロ | observer_log への記録まで完了させる |
| 「改善した」を数値なしで言う | 測定できない改善は改善ではない | Brier/エラー率/件数を必ず添える |
| KNOWN_MISTAKESへの追記だけ | パターン登録がないと再発防止できない | auto-codifier.py の実行まで確認する |
| 一時的な改善を恒久的学習と言う | 運や偶然の可能性がある | 同カテゴリで3回以上の連続成功が必要 |
| 保存したのに次回参照しなかった | タイプ5はあってもタイプ6がない | session-start.sh の引き継ぎ注入を確認する |
