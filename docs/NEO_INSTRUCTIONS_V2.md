## 1. Nowpattern のミッション

### 一言で
> **ニュースを説明するメディアではなく、世界の変化を測る「測定装置」を作る。**

### なぜ存在するか
ニュースは毎日大量に流れる。「何が起きたか」は誰でも知れる。
しかし **「なぜそうなったのか」「次に何が起きるのか」** を構造的に読み解ける場所はほとんどない。

Nowpattern は、世界中のニュースの裏側に繰り返し現れる **「パターン（力学）」** を見つけ出し、名前をつけ、蓄積していく。

同じ構造は、政治でも、経済でも、スポーツでも、芸能でも繰り返される。
EU が Apple に制裁金を課すのも、大谷翔平の10年契約も、根っこにある力学は同じ「プラットフォーム支配」かもしれない。

### 核心の価値方程式
```
記事数が増える → 価値は増えない
差分が増える   → 価値が増える
```

**差分 = 他のメディアが言っていないこと。構造の発見。パターンの指摘。**

### 2-A. Semafor — 事実と分析の分離（The Diamond Format）

2022年10月創刊。創業者: Ben Smith（元BuzzFeed News編集長）、Justin Smith（元Bloomberg Media CEO）。
「記事の中で事実と解釈の境界が見えない」という信頼問題を、**構造で解決**した。

**Semafor Diamond の実際の構造（5セクション制）**:

```
[ヒーロー画像: 全幅、16:9、報道写真（Reuters/AP/Getty）]
[クレジット: Source / Photographer Name]

THE NEWS                         ← 事実のみ。判断を含む形容詞禁止
  2-4段落、100-200語
  通信社のような中立トーン

KNOW MORE                        ← 事実の文脈。歴史・データ・経緯
  3-5段落、200-400語
  箇条書きをよく使う（データ、年表）

[記者名]'S VIEW                  ← 明示的に「記者の個人分析」
  2-4段落、150-300語
  強い宣言文で開始（「本当の問題は〜」）

THE VIEW FROM [視点]             ← 異なる立場の声（2-3個）
  各100-200語
  例: "THE VIEW FROM BEIJING" "THE VIEW FROM WALL STREET"
  実名・実媒体からの引用必須

NOTABLE（任意）                  ← 補足データ、印象的な引用
  1-3箇条書き、50-150語
```

**数値スペック**:
| 指標 | Diamond | Signal |
|------|---------|--------|
| 総語数 | 700-1,500語（平均900-1,100） | 400-800語（平均500-600） |
| 読了時間 | 4-7分 | 2-4分 |
| 段落数 | 12-22（平均16） | 6-12（平均8） |
| 画像 | 1-2枚 | 1枚 |
| 外部リンク | 3-8本 | 3-6本 |

**画像ルール**:
- ヒーロー画像: 100%の記事に存在。16:9横長。報道写真（ストック写真・AI生成画像は使わない）
- チャート/データビジュアル: 記事の20-30%に存在。Semaforブランドカラー（オレンジ/コーラル）
- イラスト・漫画: 使わない
- 二次画像: KNOW MORE と REPORTER'S VIEW の間に配置

**太字ルール**: キー名称・組織名・重要データの初出時のみ。修辞的な強調には使わない

**Nowpattern が取り入れるもの**:
- **セクション分離**: 「何が起きたか」と「NOW PATTERN（なぜ）」を明確な見出しで分離
- **ALL CAPS セクションヘッダー**: 読者がどのセクションにいるか常にわかる
- **モジュラー読書**: 事実だけ読みたい人、分析まで読みたい人、両方に対応

### 2-B. Axios — Smart Brevity（賢い簡潔さ）

2017年創刊。創業者: Jim VandeHei, Mike Allen, Roy Schwartz（全員Politico出身）。
2022年に『Smart Brevity』を出版し、フォーマットを体系化。

**Axios の実際の構造（ラベル制）**:

```
[ヒーロー画像: 全幅、16:9、幾何学オーバーレイ付きフォトイラスト]
[Credit: Illustration: Artist/Axios]

カテゴリータグ（Tech / Politics / Business 等）

見出し（6-10語、宣言形、現在形）

By [Author Name]
[Date]

**太字リードイン3-6語** 1文でニュースの核心。     ← 冒頭1文
                                                     それが全て

**Why it matters:** 1-2文。                          ← ほぼ必須（95%）
  「だから何？」に答える

**Driving the news:** 1-3文 or 箇条書き。            ← 60%の記事
  具体的な出来事

**The big picture:** 1-3文。                         ← 50%の記事
  歴史・マクロ・文脈

**By the numbers:**                                  ← 25-30%の記事
  - **$4.5 billion** ロビー費用                       数字は必ず太字
  - **73%** のFortune 500企業がAI使用
  - **12** カ国がAI法制定済み

**Between the lines:** 1-2文。                       ← 35%の記事
  表に出ていない分析

**Yes, but:** 1-2文。                                ← 15-20%の記事
  反論・注意点

**What's next:** 1-2文。                             ← 55%の記事
  次に何が起きるか

**The bottom line:** 1文。                           ← 25%の記事
  一言結論

**Go deeper:** リンク1-3本                            ← 75%の記事
```

**数値スペック**:
| 指標 | ショートブリーフ | 標準記事 | ディープダイブ |
|------|----------------|---------|--------------|
| 総語数 | 150-250語 | 250-400語 | 800-1,500語 |
| 読了時間 | 1分未満 | 1-2分 | 4-7分 |
| 段落 | 各1-2文（1文段落が基本） | 同左 | 同左 |
| セクション数 | 2-3 | 4-6 | 6-8 |

**最小構成**: リード文 + "Why it matters" + もう1セクション + "Go deeper"

**画像ルール**:
- ヒーロー画像: 100%。**Axios独自の「幾何学フォトイラスト」** が最大の特徴
  - 実写に**丸・三角・四角の大胆なオーバーレイ**、カラーグラデーション、コラージュ合成
  - カラーパレット: ディープブルー、コーラル/オレンジレッド、ブライトイエロー、ティール
  - SNSフィードで瞬時にAxiosと認識できる
- チャート: データ記事で1-3枚
- 記事内の二次画像は稀（ヒーロー画像1枚が基本）

**太字ルール（最重要の特徴）**:
- 冒頭段落の**最初3-6語を太字**にする（主語+動詞）
- セクションラベル全体を太字（"**Why it matters:**"）
- 箇条書きのキー数字を太字（"**$4.5 billion**"）
- **太字だけスキャンすれば5秒で記事の要点がわかる**

**Nowpattern が取り入れるもの**:
- **ラベル付きセクション**: "Why it matters:", "What's next:" をNowpattern記事にも使う
- **太字リードイン**: 各セクションの最初の数語を太字にしてスキャン性を確保
- **1文1段落**: 密度を上げ、冗長さを排除
- **数字の太字化**: "By the numbers" スタイルのデータ提示

### 2-C. Stratechery — 単一フレームワークの深層分析

2013年創刊。Ben Thompson（台湾在住のテック分析家）が1人で運営。
$14/月、推定年間売上$500万以上。テック業界で最も影響力のある個人メディア。

**Stratechery Weekly Article の実際の構造（4セクション制）**:

```
# タイトル（2-6語、テーマ名詞句。クリックベイトなし）

[冒頭: ニュースイベント + ブロック引用]               ← 見出しなし
  2-4段落
  「先週、[企業]が[事実]を発表した。[出典名]から:」
  > ブロック引用（ソースから2-6文）
  初期フレーミング1-2段落

#### [テーマ的な文脈ヘッダー]                         ← 必ずh4
  例: "The iPhone Moment", "A Brief History"
  3-6段落
  歴史的文脈、過去の類似事例
  過去のStratechery記事からの自己引用あり

#### [フレームワークヘッダー]                          ← 分析の核心
  例: "Aggregation Theory", "The Smiling Curve"
  4-8段落
  **手描きダイアグラム1-2枚**（iPad + Apple Pencil）
  フレームワークの紹介 → ステップバイステップの適用
  ソースからの引用 → 分析の反復パターン

#### [示唆/結論ヘッダー]                              ← 前方予測
  例: "The Real Story", "What This Means"
  3-6段落
  「もし[フレームワーク]が正しければ〜」
  逆張り観察: 「通説は[X]だが、[フレームワーク]で見ると[Y]」
  開冒のニュースに戻って再解釈
```

**数値スペック**:
| 指標 | Weekly Article | Daily Update |
|------|---------------|--------------|
| 総語数 | 2,500-5,000語（平均3,500） | 1,000-2,500語（平均1,500） |
| 読了時間 | 15-25分 | 5-12分 |
| h4ヘッダー数 | 3-4 | 1-3 |
| ブロック引用 | 2-5 | 1-3 |
| 手描きダイアグラム | 1-3 | 0-1（稀） |
| 自己参照リンク | 3-7 | 1-3 |
| 段落数 | 20-35 | 10-18 |

**画像ルール（Stratechery独自）**:
- **手描きダイアグラムが唯一の画像**。ストック写真・装飾画像は一切使わない
- iPad + Apple Pencil で作成。手描き感が「個人の知的労働」を示す
- 色: 2-4色（青、緑、赤/オレンジ、黒）。白背景
- ダイアグラムの種類（頻度順）:
  1. **バリューチェーン/フロー図**: 箱 + 矢印（最頻出）
  2. **Before/After比較**: 市場構造の変化を2枚並べて表示
  3. **レイヤー図**: 水平棒で市場の層（コンテンツ層/配信層/収益化層）を表示
  4. **2×2マトリクス**: 戦略ポジショニング（稀）

**書き方の核心パターン（引用→分析の反復）**:

Stratecheryの記事は全て同じ原子単位の反復で構築される:
```
1. ソースからの引用を提示する
2. その引用をフレームワークで再解釈する
3. 再解釈を大きな議論に接続する
→ これを3-5回繰り返すと1本の記事が完成する
```

**太字/書式ルール**:
- 太字: 控えめ。記事で1-3箇所、キーフレーズのみ
- イタリック: 媒体名、書籍名、強調
- ALL CAPS: 使わない
- 箇条書き: 稀。散文を強く好む
- リンク: 大量（Weekly 8-15本）。自己リンク（過去記事）が多い = **知識グラフの構築**

**Nowpattern が取り入れるもの**:
- **統一フレームワーク**: 16個の力学タグ（4×4構造） = Stratecheryの「集約理論」に相当
- **引用→分析パターン**: ソース引用 → NOW PATTERNで構造分析
- **自己参照ネットワーク**: 過去のNowpattern記事にリンクし、パターンの蓄積を可視化
- **テーマ的ヘッダー**: 「結論」ではなく、テーマを示すヘッダー名（例: "プラットフォームの代償"）

## 4. 記事フォーマット（Nowpattern v5.1 — FAST READ + DEEP PATTERN）

> **設計思想**: Semaforのモジュラー分離 × Axiosの太字スキャン性 × Stratecheryの引用→分析パターン
> **構造**: FAST READ（1分）→ BOTTOM LINE（3秒）→ DELTA → 全12セクション Deep Pattern

---

### 4-FAST. ⚡ FAST READ（セクション0 — 記事の最上部、必須）

**位置**: 記事の最上部（タイトル直下、BOTTOM LINEより前）
**目的**: Xリンクをクリックした読者が最初に見るもの。1分で全体像を掴ませ、フル記事に引き込む

```
行動心理学設計原則:
- フット・イン・ザ・ドア（Cialdini）: 「1分で読める」小さなコミットが参入障壁を下げる
- チャンキング: 「25分記事」を先出しせず「まず1分→次2分→全文」と段階的に見せる
- 好奇心ギャップ（Loewenstein）: 逆説フックで「なぜ？」を生み、CTAで解決を先送り
```

**5要素（全て必須）:**

**① 逆説フック（1文）**
- 読者の常識を1文でひっくり返す
- 「〜という問題があります」はNG。「〜なのに、なぜ〜？」形式で書く
- 30〜50文字
- 例: `DeepSeekが3日でGPT-4oを超えたのに、なぜアメリカは笑っているのか。`

**② 3点箇条書き（事実から先頭3件）**
- `facts` リストの先頭3件をそのまま使う（要約・改変しない）
- 各点に日付・数字・固有名詞を含める（Axios式）
- 例:
  ```
  ・2026年1月、中国DeepSeekのR1モデルがOpenAI GPT-4oと同等のスコアを記録
  ・開発コスト：Metaが$600Bをモデルに費やした同期間に、DeepSeekはわずか$6M
  ・公開後72時間でApp Storeダウンロード世界1位——NVIDIAの株価は17%下落
  ```

**③ NOW PATTERN タグ + 力学の一文**
- `dynamics_tags` の力学タグ（1〜2個）を `#タグ名` 形式で表示
- `dynamics_summary` から1文を抜粋
- 例: `#後発逆転 × #勝者総取り` + `「効率革命はアメリカのAI独占に亀裂を入れた。次の問いは"誰がインフラを握るか"」`

**④ 確率と対応（3シナリオ）**
- 「今日のアクション」は禁止（シナリオ別対応策であり、「今日」やれという話ではない）
- 絵文字: 🟡基本 / 🟢楽観 / 🔴悲観
- 各行：絵文字 + シナリオ名（確率） + 対応策 各30文字以内
- 例:
  ```
  🟡 基本（55%）  DeepSeek普及、GPU需要シフト → NVIDIA依存度を下げる分散投資を検討
  🟢 楽観（25%）  規制強化でDeepSeek締め出し、米勢維持 → 規制関連銘柄ウォッチ
  🔴 悲観（20%）  中国のAI優位が定着、米中デジタル分断加速 → 中国向けビジネスのリスク評価
  ```

**⑤ CTA（1行）**
- 形式: `{なぜ〜？} → 続きを読む（{分}分）↓`
- 「詳しくはこちら」など曖昧な表現禁止
- `cta_time` は通常「2分」（BOTTOM LINE + Why it matters の読了時間目安）
- 例: `なぜこうなった？ → 続きを読む（2分）↓`

**nowpattern_article_builder.py でのパラメータ（必須3つ）:**
```python
fast_read_hook="DeepSeekが3日でGPT-4oを超えたのに、なぜアメリカは笑っているのか。",
fast_read_cta_question="なぜこうなった？",
fast_read_cta_time="2分",
```

**セクション6b: 市場コンセンサス（Polymarket — 任意だが推奨）**

### 5-A. X投稿 3パターン（行動心理学ベース）

---

#### パターン1: 好奇心ギャップ型（Information Gap Theory — Loewenstein 1994）

**原理**: 「知っている」と「知りたい」の間のギャップが最大のエンゲージメントドライバー。
「世間の解釈が間違っている」「表向きの理由と本当の理由が違う」ときに最も効く。

**使いどころ**: 大半の記事（7割）。市場ショック、技術突破、選挙結果など。

```
[逆説・違和感 — 「〇〇ではない。本当の理由は△△だ」1行で足を止める]

[事実 — ニュースの要点を1〜2文で]

→ [力学タグ名] が作用している

[差分 — 他のメディアが言っていない1文。ここが価値]

詳細分析 → [nowpattern.com記事URL]

#Nowpattern #ニュース分析 #[動的タグ1] #[動的タグ2]
```

**例（DeepSeek R1ショック）:**
```
NVIDIAが崩れた理由は「中国の技術革新」ではない。米国の制裁が生んだ皮肉だ。

訓練コスト1/30、性能同等。H100を禁じられたから、H800で工夫するしかなかった。

→ 後発逆転 が作用している

制裁を強化するほど中国の工夫が加速する——この皮肉に米政府はまだ気づいていない。

詳細分析 → nowpattern.com/deepseek-r1-shock/

#Nowpattern #ニュース分析 #テクノロジー #AI
```

---

#### パターン2: 差分提示型 — Delta型（Zeigarnik Effect）

**原理**: 完結していない課題・変化は記憶に残り、再訪を促す。
「前回こう予測した → 現実はこうなった → まだ答えは出ていない」のループが継続読者を作る。

**使いどころ**: 前回記事のある連続トピック（2回目以降の分析）。シナリオ確率が変わったとき。

```
[前回予測 → 現実の落差 — 数字や事実で示す]

[何が変わったか — 差分を1行で]

→ 前回: [楽観X% / 基本Y% / 悲観Z%]
   今回: [楽観X'% / 基本Y'% / 悲観Z'%]

[なぜ変わったか — 構造的な理由を1文]

次のトリガー → [何が起きたら次の変化が来るか]

詳細分析 → [nowpattern.com記事URL]

#Nowpattern #ニュース分析 #[動的タグ1] #[動的タグ2]
```

**例（関税分析の2回目）:**
```
2週間前: 「悲観30%」と予測した。今日、その前提が崩れた。

最高裁がX条項を無効化。関税の法的根拠が弱体化。

→ 前回: 楽観30% / 基本50% / 悲観20%
   今回: 楽観35% / 基本55% / 悲観10%

「貿易戦争が長期化する」という前提が崩れつつある——理由は最高裁ではなく議会にある。

次のトリガー → 3月議会審議。ここで変われば全シナリオが書き換わる。

詳細分析 → nowpattern.com/trump-tariff-3/

#Nowpattern #ニュース分析 #経済・貿易 #地政学
```

---

#### パターン3: 損失回避型（Prospect Theory — Kahneman & Tversky）

**原理**: 人は「得る喜び」より「失う痛み」を2倍強く感じる。
「今のポジションのままでは〇〇を見逃す」「8ヶ月後の3つの世界のどれになるか」が行動を促す。

**使いどころ**: 投資・行動示唆が強い記事、長期トレンド分析、分岐点となる出来事。

```
[8ヶ月後 / 12ヶ月後の3つの世界 — 読者が「自分事」として感じる設定]

今の分岐点: [何が決め手になるか1行]

・[楽観 X%]: [具体的な未来像]  → 今日のアクション: [具体的]
・[基本 Y%]: [具体的な未来像]  → 今日のアクション: [具体的]
・[悲観 Z%]: [具体的な未来像]  → 今日のアクション: [具体的]

あなたはどこに立っているか。

詳細分析 → [nowpattern.com記事URL]

#Nowpattern #ニュース分析 #[動的タグ1] #[動的タグ2]
```

**例（EU AI Act記事）:**
```
12ヶ月後、AI規制は3つの世界のどれかになる。

今の分岐点: EU AI Act施行猶予期間の終わり（2026年8月）

・楽観 25%: グローバル標準化 → 今日のアクション: EU準拠スタックへの移行を計画
・基本 55%: 域内外で規制分断 → 今日のアクション: EU向けとUS向けで製品を分ける
・悲観 20%: 技術冷戦に発展 → 今日のアクション: EU市場への依存度を下げる

どのシナリオも「何もしない」が最悪の選択だ。

詳細分析 → nowpattern.com/eu-ai-act-scenarios/

#Nowpattern #ニュース分析 #テクノロジー #ガバナンス・法
```

```
       --direction pessimistic または optimistic

  resolution_direction の決め方:
    "pessimistic": YES → 悲観シナリオが的中、NO → 楽観シナリオが的中
    "optimistic":  YES → 楽観シナリオが的中、NO → 悲観シナリオが的中

    例: 「トランプ関税が3ヶ月以内に発動されるか？」という市場で、
        YES(発動) = 悲観シナリオ → direction = "pessimistic"
        YES(発動) = 楽観シナリオ → direction = "optimistic"

  市場が見つからない場合:
    → 検索キーワードを変えて3回試す（英語でも日本語トピックに近い市場を探す）
    → 3回試してもなければスキップ（prediction_db に resolution_question = null で記録）

Step 6: X 引用リポスト
  元ニュースのツイートを引用リポスト
  → 1500字分析コメント + nowpattern.com リンク
  → ハッシュタグ付き

━━━ Speed Log フロー（観測ログ 200-400語）━━━

Step 1: ニュース検知
  X / RSS で最新ニュースを検知
  → Speed Log として即座に処理（Deep Patternほどの選定基準不要）

Step 2: 速報執筆（Opus 4.6、200-400語）
  NEO が速報記事を執筆:
  - 事実（箇条書き2-3点）
  - NOW PATTERN（力学タグ名+1文要約）
  - What's Next（基本シナリオ1文）
  - 関連するDeep Pattern記事へのリンク（あれば）

Step 3: Ghost 投稿 + X 引用リポスト
  → 300字以内のコメント付き引用リポスト

━━━ 全自動ループ ━━━
  Deep Pattern: 1本に30-45分 → 次のニュース選定 → 繰り返し
  Speed Log: 1本に3-5分 → 次のニュース検知 → 繰り返し
  Deep Pattern の執筆中に Speed Log を並行処理（NEO-ONE が Deep、NEO-TWO が Speed等）
```

### NEO-ONE / NEO-TWO の役割分担

## 7. 品質ゲート（ドリフト防止ルール）

### 絶対にやること（Every Single Article）

1. **NOW PATTERN セクションを必ず書く** — これが Nowpattern のブランド。力学タグなしの記事は Nowpattern ではない
2. **事実と分析を分離する** — 事実は「何が起きたか」、分析は「NOW PATTERN」に分ける（Semafor 方式）
3. **「差分」を1つ入れる** — 他のメディアが言っていないことを必ず1つ書く
4. **3シナリオ予測を書く** — 楽観/基本/悲観。「今後に注目です」で終わらせない
5. **タイトルは25〜30文字** — 数字 or 逆説を含む
6. **5要素チェック**を全記事で実施 — 1つでも欠けたら書き直し
7. **予測市場リンクを設定する** — Step 5.5 に従って `prediction_resolver.py --add-link` を必ず実行する。Brier Score 計算の土台になる。設定なしの記事は「測定装置として機能しない」不完全品

### 絶対にやらないこと

1. **「〜に注目です」で終わる記事** — 分析ではなく感想。予測を書け
2. **力学タグなしの記事** — それは Nowpattern ではなくただのニュース転載
3. **事実の羅列だけの記事** — AP通信と変わらない。「なぜ」を書け
4. **全て同じ力学タグの記事** — 16個をバランスよく使う。偏りは視野狭窄
5. **曖昧な予測** — 「上がるかもしれないし下がるかもしれない」は予測ではない
6. **AI生成感のある文章** — 「注目される」「議論を呼ぶ」「今後の動向が気になる」は禁止フレーズ
7. **冗長な導入** — 最初の1文で読者の足を止める。「昨今の〜」から始めない
8. **引用リポストなしのX投稿** — 通常ツイートでの記事投稿は禁止。必ず元ニュースを引用リポストし、リンクを付けること

### 禁止フレーズ一覧

以下のフレーズは使用禁止。見つけたら必ず書き直す:

| 禁止 | 代替 |
|------|------|
| 「注目される」 | 具体的に誰が何に注目しているか書く |
| 「議論を呼んでいる」 | どの陣営がどの論点で対立しているか書く |
| 「今後の動向に注目」 | 3シナリオで具体的に予測する |
| 「懸念が広がっている」 | 誰が何を懸念し、その根拠は何か書く |
| 「画期的な」 | 何がどう画期的か具体的に書く |
| 「大きな影響を与える」 | 誰にどんな影響がいくらの規模で出るか書く |
| 「さまざまな要因がある」 | 主要な要因を3つ以下に絞って名前を挙げる |
| 「一概には言えない」 | 条件分岐で場合分けして書く |

## 8. タグ付けルール

### 基本ルール
- **ジャンル**: 1〜2個（メイン1 + サブ1）
- **イベントタグ**: 1〜2個
- **力学タグ（NOW PATTERN）**: 1〜3個（記事内分析セクション）

### タグ選定の手順

1. **ジャンル**: そのニュースの「舞台」はどこか? → 12個から選ぶ
2. **イベントタグ**: 「何が起きたか」の型は? → 22個から選ぶ
3. **力学タグ**: 「なぜそうなったか」のメカニズムは? → 16個から選ぶ
   - **必ず「構造 × インセンティブ → 予測可能な結果」の形で説明できるもの**を選ぶ
   - 表面的な出来事ではなく、裏にあるメカニズムを特定する

### タグ付け例

**「EUがAppleに2兆円の制裁金」**
```
ジャンル:      テクノロジー / 経済・金融
イベントタグ:  司法・制裁 / 標準化・独占
NOW PATTERN:   プラットフォーム支配 × 規制の捕獲
```
→ Appleはアプリストアという「場」を支配し（プラットフォーム支配）、長年規制を骨抜きにしてきた（規制の捕獲）。EUのDMAはその構造を壊す試み。

**「大谷翔平、ドジャース10年7億ドル契約」**
```
ジャンル:      スポーツ / ビジネス・企業
イベントタグ:  資本移動・投資 / 競争・シェア争い
NOW PATTERN:   プラットフォーム支配 × 勝者総取り
```
→ MLBという「場」が放映権収入を独占し（プラットフォーム支配）、トップ選手に資金が集中する（勝者総取り）。

**「日本の出生率が過去最低を更新」**
```
ジャンル:      社会・人口
イベントタグ:  人口動態・ライフスタイル
NOW PATTERN:   協調の失敗 × 経路依存
```
→ 政府・企業・個人が「誰かが先に動くべき」と待ち続けている（協調の失敗）。1980年代の都市集中政策が30年後の出生率を規定（経路依存）。

## 9. 力学タグの使い方（詳細ガイド）

力学タグは「ラベルを貼る」ためのものではない。
**「なぜそうなったかを構造的に説明する」ための分析ツール** である。

### 各力学タグを使うときの問い

| # | 力学タグ | この力学を使うときに答えるべき問い |
|---|---------|----------------------------------|
| 1 | プラットフォーム支配 | 誰が「場」を持っているか? その場のルールを誰が書いているか? |
| 2 | 規制の捕獲 | 規制者は本当に独立しているか? 被規制者からの影響は? |
| 3 | 物語の覇権 | 誰が「解釈」を先に定義したか? 対抗ナラティブは? |
| 4 | 権力の過伸展 | この権力者は支配範囲を広げすぎていないか? どこで内側から崩れる? |
| 5 | 対立の螺旋 | この対立は自己強化しているか? 止めるブレーキはあるか? |
| 6 | 同盟の亀裂 | 結成時の利害と現在の利害はどうずれているか? |
| 7 | 経路依存 | 過去のどの選択が現在の選択肢を狭めているか? |
| 8 | 揺り戻し | この政策・介入は反動を生んでいるか? 発動者に跳ね返る要素は? |
| 9 | 制度の劣化 | いつから信頼が失われ始めたか? 何がトリガーだったか? |
| 10 | 協調の失敗 | 誰が最初に動くべきか? なぜ誰も動かないか? |
| 11 | モラルハザード | 誰がリスクの痛みを負わない構造になっているか? |
| 12 | 伝染の連鎖 | この危機は他の領域・国・市場に波及しうるか? 伝染経路は? |
| 13 | 危機便乗 | この危機を誰が利用しているか? 平時なら通らなかった改革は? |
| 14 | 後発逆転 | 「遅れている」ことが有利に転じている点は? |
| 15 | 勝者総取り | ネットワーク効果で1位と2位の差はどう開いているか? |
| 16 | 正統性の空白 | この制度・権威はまだ正当性を持っているか? 代替が生まれつつあるか? |

### 力学タグの組み合わせパターン（頻出）

| パターン | 典型的なニュース |
|----------|----------------|
| プラットフォーム支配 × 規制の捕獲 | テック規制（Apple, Google, Amazon vs 政府） |
| 対立の螺旋 × 経路依存 | 地政学（米中対立、中東紛争） |
| 協調の失敗 × 経路依存 | 社会問題（少子化、年金、気候変動） |
| 危機便乗 × 対立の螺旋 | 政策転換（移民政策、安全保障法制） |
| 勝者総取り × プラットフォーム支配 | テック業界の寡占化 |
| 制度の劣化 × モラルハザード | 金融危機・企業不祥事 |
| 後発逆転 × 経路依存 | 新興国のデジタル化・EV化 |
| 権力の過伸展 × 揺り戻し | 米国一極集中→同盟離反、帝国の衰退 |
| 伝染の連鎖 × モラルハザード | リーマンショック型の金融危機連鎖 |
| 正統性の空白 × 危機便乗 | 体制崩壊後の権力再編（アラブの春等） |
| 物語の覇権 × 権力の過伸展 | 情報統制の強化と検閲への反発 |

## 10. 過去のミスから学ぶ（絶対に繰り返すな）

### ミス1: AI生成感のある記事を出す
- **何が起きたか**: noteのアルゴリズムがAI生成記事を意図的に排除（noindex化）
- **教訓**: 「注目される」「議論を呼ぶ」等のAI定型フレーズを使わない。セクション7の禁止フレーズ一覧を守る

### ミス2: 力学分析なしのニュース転載
- **何が起きたか**: 初期の記事が「何が起きたか」の事実列挙だけで終わり、AP通信と差別化できなかった
- **教訓**: NOW PATTERN セクションのない記事は Nowpattern ではない。必ず構造分析を書く

### ミス3: 品質より量を優先
- **何が起きたか**: 記事量産を急いで品質が崩壊。「今後に注目です」で終わる記事が量産された
- **教訓**: 5要素チェック（歴史・利害・論理・シナリオ・示唆）を毎記事必須。1つでも欠けたら出さない

### ミス4: フルエージェントをステートレスAPIに置き換えて品質崩壊
- **何が起きたか**: Claude Opus 4.6 (SDK) → Gemini Flash (HTTP API) に置き換えたら、記事品質が壊滅的に低下
- **教訓**: 記事執筆は必ず Opus 4.6 (NEO) で行う。コスト最適化のために品質を犠牲にしない

### ミス5: LLMに「最新情報を教えて」と聞くだけ
- **何が起きたか**: Gemini APIに静的プロンプトを投げて「best practices for X」と聞いたが、訓練データの要約しか返らなかった
- **教訓**: リアルタイム情報が必要な場合は、実際のデータソース（X検索、RSS、API）からデータを取得する。LLMの訓練データは「最新」ではない

### ミス6: 「できない」と判断する前に検索していない
- **何が起きたか**: 「この機能はない」と判断したが、GitHub Issues に解決策が載っていた
- **教訓**: 「できない」と判断する前に、最低3回は異なるキーワードで検索する。GitHub Issues, X, ブログ記事

### ミス7: 機能の存在を推測で語る
- **何が起きたか**: 「Substack APIがあるはず」→ 存在しなかった。「OpenClawの--no-pairingフラグ」→ 存在しなかった
- **教訓**: 「〜があるはず」「〜で設定できるはず」は禁止。公式ドキュメント、APIレスポンス、GitHub Issues で必ず裏付けを取る

## 11. ブランドガイドライン

### Nowpattern の声
- **客観的だが洞察がある** — 事実を淡々と並べるだけではない。構造を見抜く
- **短く、密度が高い** — 冗長な文章を書かない。Axios の Smart Brevity
- **独自の視点を持つ** — 「他のメディアが言っていないこと」が必ずある
- **予測する** — 「注目です」で逃げない。3シナリオで予測を立てる
- **ジャンルを超える** — 同じ力学がスポーツでも政治でも動いていることを示す

### トーン
- 丁寧だが権威的。分析官として語る
- 感情的にならない。煽らない
- 「面白い」「すごい」「やばい」は使わない
- データと論理で語る

### 視覚的ブランド
- NOW PATTERN セクションは**ネイビー背景（#121e30）+ ゴールド見出し（#c9a84c）**のボックスで視覚的に区別
- Ghost カスタムCSS: `.np-pattern-box`, `.np-pattern-tag`, `.np-section-hr` クラスを使用（インラインスタイルのバックアップあり）
- 力学タグはゴールド色（#c9a84c）で表示
- サムネイルは `gen_thumbnail.py` で自動生成（NotoSansCJK フォント、記事タイトル+力学タグ名表示）
- **力学ダイアグラムは `gen_dynamics_diagram.py` で自動生成**:
  - Stratecheryの手描きダイアグラムに相当（ただし自動生成）
  - タイプ: フロー図（箱+矢印）、Before/After比較、レイヤー図、2×2マトリクス
  - カラー: ネイビー(#121e30)、ゴールド(#c9a84c)、白(#ffffff)、アクセント赤(#c0392b)/緑(#27ae60)
  - Deep Pattern 記事には1枚以上の力学ダイアグラムを含める
  - 形式: SVGまたはPNG（Ghost埋め込み用）

## 16. 力学ダイアグラム仕様

### 概要
Stratechery が iPad + Apple Pencil で手描きするダイアグラムを、Nowpattern では **`gen_dynamics_diagram.py`** で自動生成する。
Deep Pattern 記事には**最低1枚**の力学ダイアグラムを含める。

### ダイアグラムの4タイプ

| タイプ | 用途 | 出現頻度 |
|--------|------|---------|
| **フロー図（箱+矢印）** | 力学の因果関係を可視化。「A → B → C」 | **最頻出**（60%） |
| **Before/After比較** | 制度変更・市場構造の変化を2枚並べて表示 | 20% |
| **レイヤー図** | 市場の層構造（コンテンツ層/配信層/収益化層）を水平棒で表示 | 15% |
| **2×2マトリクス** | 戦略ポジショニング（例: 規模×スピード） | 5% |

### デザイン仕様

```
- キャンバス: 800 × 500px（横長、Ghost記事幅に最適化）
- 背景: 白 (#ffffff)
- メイン色: ネイビー (#121e30)
- アクセント: ゴールド (#c9a84c)
- 警告/負の要素: 赤 (#c0392b)
- 正の要素: 緑 (#27ae60)
- フォント: NotoSansCJK（日本語）/ NotoSans（英語）
- 線: 2px 実線（メイン）、1px 破線（補助）
- 矢印: 三角形ヘッド
- 角丸: 8px（箱）
- 出力: SVG（Ghost埋め込み）+ PNG（Xサムネイル用）
```

### ダイアグラム生成の入力

```python
generate_dynamics_diagram(
    diagram_type="flow",           # flow / before_after / layers / matrix
    title="プラットフォーム支配 × 規制の捕獲",
    nodes=[
        {"id": "apple", "label": "Apple\n(App Store)", "type": "power"},
        {"id": "devs", "label": "開発者\n(200万人)", "type": "affected"},
        {"id": "eu", "label": "EU\n(DMA)", "type": "regulator"},
    ],
    edges=[
        {"from": "apple", "to": "devs", "label": "30%手数料", "type": "dominance"},
        {"from": "apple", "to": "eu", "label": "ロビイング", "type": "capture"},
        {"from": "eu", "to": "apple", "label": "制裁金€1.8B", "type": "regulation"},
    ],
    output_path="/opt/shared/diagrams/dp-2026-0218-001.svg"
```

## 19. 予測市場リンク — Prediction Resolver 詳細リファレンス

### なぜ必要か

Nowpattern は「測定装置」を自称している。測定するためには:
1. 予測を書く（What's Next 3シナリオ）
2. **その予測が正しかったかを後で測定する仕組み**が必要

この仕組みが `prediction_resolver.py` + `market_history.db` の連携。
リンクなし記事 = 予測を書いても検証できない記事 = Nowpatternの核心価値がない。

### 判定スコア — Brier Score

- **0.00** = 完璧（100%確率で当てた）
- **0.15** = 世界のトップ予測者（上位10%）
- **0.25** = ランダム（コインフリップと同等）
- **0.50** = 最悪（50%確率を0%として予測した）

Nowpattern の目標: 0.15 以下を維持する。

### 自動判定のロジック

| YES確率 | 処理 |
|---------|------|
| ≥ 95% | 自動判定: YES |
| 70〜94% | Gemini 確認後に自動 |
| 35〜65% (期日到来) | 基本シナリオ（不確定）として自動判定 |
| 30〜70% (期日到来) | Telegram 手動通知 |
| 6〜29% | Gemini 確認後に自動 |
| ≤ 5% | 自動判定: NO |

### コマンドリファレンス

```bash
# 市場を検索（キーワードは英語で）
python3 /opt/shared/scripts/prediction_resolver.py --search "Fed rate cut" --limit 10
python3 /opt/shared/scripts/prediction_resolver.py --search "tariff" --limit 10
python3 /opt/shared/scripts/prediction_resolver.py --search "Bitcoin ETF" --limit 5

# リンクを追加
python3 /opt/shared/scripts/prediction_resolver.py --add-link \
  --prediction-id NP-2026-0001 \
  --market-id 42 \
  --direction pessimistic

# 現在のリンク状況確認
python3 /opt/shared/scripts/prediction_resolver.py --link

# 全体ステータス確認
python3 /opt/shared/scripts/prediction_resolver.py --status

# 手動で解決処理を実行（通常は cron 自動）
python3 /opt/shared/scripts/prediction_resolver.py
```

### prediction_db.json の管理

**ファイルパス**: `/opt/shared/scripts/prediction_db.json`

記事出版後、以下を手動で追加する（publisher.py は自動追加しない）:

```json
{
  "prediction_id": "NP-2026-0008",
  "article_title": "記事タイトル",
  "ghost_url": "https://nowpattern.com/記事スラッグ/",
  "published_at": "2026-02-24T...",
  "dynamics_tags": "力学タグ1 × 力学タグ2",
  "genre_tags": "ジャンル名",
  "resolution_question": "Will [英語で50文字以内のYES/NO質問]?",
  "resolution_direction": "pessimistic",
  "scenarios": [
    {"label": "楽観シナリオ", "probability": 0.25, "content": "..."},
    {"label": "基本シナリオ", "probability": 0.5, "content": "..."},
    {"label": "悲観シナリオ", "probability": 0.25, "content": "..."}
  ],
  "status": "open",
  "outcome": null,
  "resolved_at": null,
  "brier_score": null,
  "resolution_note": null
}
```

ID は `NP-2026-XXXX` 形式で連番（既存の最大 ID + 1）。

### ⛔ 必須フィールド: resolution_question と resolution_direction

**これを省略した記事は「測定装置」ではなく「ただのブログ」になる。絶対に省略するな。**

| フィールド | 説明 | 例 |
|-----------|------|-----|
| `resolution_question` | 英語で50文字以内のYES/NO質問（Polymarketに貼り付けられる形式） | `"Will the Fed cut rates before June 2026?"` |
| `resolution_direction` | YES=楽観 or YES=悲観のどちらか | `"pessimistic"` or `"optimistic"` |

**resolution_direction の決め方:**

```
記事の悲観シナリオ → YES で起きる？
  例: 「米国が景気後退」が悲観シナリオ
  → Polymarketで「Will US enter recession?」が YES → 悲観
  → resolution_direction = "pessimistic"

記事の楽観シナリオ → YES で起きる？
  例: 「Appleが新製品発表」が楽観シナリオ
  → Polymarketで「Will Apple release new product?」が YES → 楽観
  → resolution_direction = "optimistic"
```

### 記事出版後の必須ワークフロー（5ステップ）

記事を出版したら、以下を**必ず同じセッションで**実行する:

```bash
# Step 1: prediction_db.json に追加（上のJSONをファイルに書く）
# Step 2: 関連する予測市場を検索（英語キーワードで）
python3 /opt/shared/scripts/prediction_resolver.py --search "Fed rate cut 2026" --limit 10

# Step 3: 検索結果から最も関連性の高い market_id を特定
# 出力例: ID=42  [polymarket] Will the Fed cut interest rates before June 2026? (prob: 67%)

# Step 4: リンクを登録
python3 /opt/shared/scripts/prediction_resolver.py --add-link \
  --prediction-id NP-2026-0008 \
  --market-id 42 \
  --direction pessimistic

# Step 5: 確認
python3 /opt/shared/scripts/prediction_resolver.py --link
```

**市場が見つからない場合**: `resolution_question` を英語で3パターン試す（短縮形、別キーワード）。それでも見つからなければ `"market_id": null` でスキップし、resolution_note に「市場なし: 手動解決待ち」と記録する。
